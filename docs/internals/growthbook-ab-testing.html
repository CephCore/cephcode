GrowthBook A/B Testing System

Runtime feature flags and progressive rollout

Source: ccb.agent-aura.top

Why Runtime A/B Testing?

Build-time feature() is all-or-nothing — either every user has it or nobody does. But product teams need finer control:

  • Roll out a feature to only 5% of users
  • Differentiate experiences by subscription tier (Free / Pro / Team)
  • Silently enable experimental capabilities for specific organizations
  • Remotely disable a broken feature at any time without a release

That's where GrowthBook comes in — a runtime, user-attribute-based feature gating and A/B testing system.

Integration Architecture

Full implementation at src/services/analytics/growthbook.ts (1,258 lines). The workflow:

1. Fetch remote config on startup

When the CLI starts, the GrowthBook SDK fetches current feature config and experiment group assignments from https://api.anthropic.com/. Uses remoteEval: true — group calculation happens server-side, the client only receives results.

2. Compute user attributes

The SDK collects current user attributes (device ID, subscription type, organization UUID, etc.) to determine which experiment group the user falls into.

3. Cache locally

Results are cached in ~/.claude.json under cachedGrowthBookFeatures. Refresh interval: 20 minutes for Anthropic employees, 6 hours for external users.

4. Query flags in code

Business code queries feature status using tengu_*-prefixed flag names. GrowthBook SDK returns the current user's group assignment value.

Naming Culture: The tengu_* World

All runtime flags use the tengu_ prefix — "Tengu" (a mythical Japanese creature) is the internal codename for Claude Code. Flag names follow an animal / plant / mineral + adjective convention, deliberately kept opaque.

tengu_kairos — Kairos Assistant Mode

Runtime toggle for the KAIROS feature. Even if build-time feature('KAIROS') passes, this flag still needs to hit for activation. Dual gating enables phased feature rollout.

tengu_amber_stoat — Explore Agent A/B Test

Controls behavioral variants of the built-in Explore sub-agent. "Amber stoat" is a randomly generated codename, unrelated to the feature's purpose — preventing feature discovery through flag names.

tengu_auto_background_agents — Background Agent Automation

Controls whether certain tasks are automatically dispatched to background agents instead of blocking the user in the foreground.

tengu_onyx_plover — Auto-Dream Background Memory

Controls the "auto-dream" feature — organizing and consolidating agent memory in the background during idle time. Another opaque codename.

tengu_glacier_2xr — Tool Search Behavior

Controls behavioral variants of Tool Search, likely an A/B test for search algorithms or ranking strategies.

tengu_birch_trellis — Bash Permission Policy

Controls the policy variant for BashTool permission decisions — possibly testing more lenient or stricter permission rules.

tengu_scratch — Scratchpad Feature

Controls an experimental "scratchpad" feature, possibly giving the AI a temporary staging area when handling complex tasks.

tengu_quartz_lantern — Diff Calculation Strategy

Controls the diff calculation approach for file writes and edits. Likely A/B testing different diff algorithms for UX impact.

Ant-Only Override Mechanism

Anthropic employees have two ways to bypass GrowthBook's remote evaluation:

Environment Variable Override

# Only works in builds where USER_TYPE=ant
CLAUDE_INTERNAL_FC_OVERRIDES='{"tengu_kairos": true}' claude

Pass a JSON object via CLAUDE_INTERNAL_FC_OVERRIDES to directly override any flag's value.

Config UI Override

In internal builds, the /config command's Gates tab provides a graphical flag management interface that can toggle any GrowthBook flag in real-time.

Experiment Tracking

GrowthBook integrates full experiment exposure tracking:

  • Each flag query logs an experiment exposure event
  • Reported via protobuf-format GrowthbookExperimentEvent
  • Includes variation_id (0 = control, 1+ = treatment) and in_experiment flag
  • Data is used to analyze causal impact of features on user behavior

GrowthBook is currently migrating from Statsig — the codebase still retains compatibility layers like checkStatsigFeatureGate_CACHED_MAY_BE_STALE().

GrowthBook A/B 测试体系

运行时功能发布与渐进式发布

来源: ccb.agent-aura.top

为什么需要运行时 A/B 测试

构建时 feature() 是"全有或全无"的——要么所有用户都有，要么所有用户都没有。但产品团队需要更精细的控制：

  • 只对 5% 的用户灰度发布新功能
  • 按订阅类型（Free / Pro / Team）差异化体验
  • 对特定组织静默开启实验性能力
  • 随时远程关闭出问题的功能，无需发版

这就是 GrowthBook 的用武之地——一个运行时的、基于用户属性的功能门控和 A/B 测试系统。

集成架构

完整实现位于 src/services/analytics/growthbook.ts（1258 行）。工作流程如下：

1. 启动时获取远程配置

CLI 启动时，GrowthBook SDK 通过 https://api.anthropic.com/ 的 API 端点获取当前的功能配置和实验分组规则。使用 remoteEval: true 模式——在服务端计算分组，客户端只拿结果。

2. 计算用户属性

SDK 收集当前用户的属性（设备 ID、订阅类型、组织 UUID 等），用于决定该用户属于哪些实验的哪个分组。

3. 缓存到本地

计算结果缓存到 ~/.claude.json 的 cachedGrowthBookFeatures 字段。刷新间隔：Anthropic 员工 20 分钟，外部用户 6 小时。

4. 代码中查询 flag

业务代码通过 tengu_* 前缀的 flag 名查询功能状态，GrowthBook SDK 返回当前用户的分组值。

这套定向系统意味着 Anthropic 可以做非常精细的实验——比如"只对 Mac 上的 Pro 订阅用户的 10% 开启新功能"。

所有运行时 flag 都以 tengu_ 为前缀——"Tengu"（天狗）是 Claude Code 的内部项目代号。flag 名采用 动物/植物/矿物 + 形容词 的命名约定，刻意保持不透明。

控制 KAIROS 功能的运行时开关。即使构建时 feature('KAIROS') 通过，仍需此 flag 命中才能激活。双重门控确保新功能可以分阶段发布。

tengu_onyx_plover — Auto-Dream 后台记忆

控制"自动做梦"功能——在空闲时后台整理和巩固 Agent 的记忆。又是一个不透明代号。

tengu_glacier_2xr — 工具搜索行为

控制 Tool Search 的行为变体，可能是搜索算法或排序策略的 A/B 测试。

tengu_birch_trellis — Bash 权限策略

控制 BashTool 权限判断的策略变体——可能在测试更宽松或更严格的权限规则。

tengu_scratch — 草稿本功能

控制一个实验性的"草稿本"功能，可能是让 AI 在处理复杂任务时使用中间暂存区。

tengu_quartz_lantern — Diff 计算策略

控制文件写入和编辑时的 diff 计算方式。可能在 A/B 测试不同的 diff 算法对用户体验的影响。

通过 CLAUDE_INTERNAL_FC_OVERRIDES 环境变量传入 JSON 对象，直接覆盖任意 flag 的值。

Config 界面覆盖

实验追踪

GrowthBook 集成了完整的实验曝光追踪：

  • 每次查询 flag 时记录实验曝光事件
  • 通过 protobuf 格式的 GrowthbookExperimentEvent 上报
  • 包含 variation_id（0=对照组，1+=实验组）和 in_experiment 标记
  • 数据用于分析功能对用户行为的因果影响

ระบบ A/B Testing ด้วย GrowthBook

การเปิดตัวฟีเจอร์แบบไดนามิกระดับ Runtime

ที่มา: ccb.agent-aura.top

ทำไมต้อง Runtime A/B Testing?

feature() แบบ build-time เป็นแบบ"ทั้งหมดหรือไม่มีเลย" — ผู้ใช้ทุกคนได้ฟีเจอร์หรือไม่มีใครได้เลย แต่ทีมผลิตภัณฑ์ต้องการการควบคุมที่ละเอียดกว่านั้น:

  • เปิดตัวฟีเจอร์ให้ผู้ใช้เพียง 5%
  • แยกประสบการณ์ตามระดับสมาชิก (Free / Pro / Team)
  • เปิดความสามารถทดลองให้เฉพาะบางองค์กรอย่างเงียบ ๆ
  • ปิดฟีเจอร์ที่มีปัญหาได้จากระยะไกลโดยไม่ต้องออกรีลีสใหม่

นี่คือจุดที่ GrowthBook เข้ามา — ระบบ gate และ A/B testing ที่ทำงานระดับ runtime โดยพิจารณาจากคุณสมบัติผู้ใช้

สถาปัตยกรรม Integration

การติดตั้งเต็มรูปแบบอยู่ที่ src/services/analytics/growthbook.ts (1,258 บรรทัด) ขั้นตอนการทำงาน:

1. ดึงการกำหนดค่าระยะไกลเมื่อเริ่มต้น

เมื่อ CLI เริ่มทำงาน GrowthBook SDK จะดึงการกำหนดค่าฟีเจอร์และกฎการจัดกลุ่มการทดลองปัจจุบันจาก API ที่ https://api.anthropic.com/ ใช้โหมด remoteEval: true — คำนวณกลุ่มที่ฝั่งเซิร์ฟเวอร์ ไคลเอนต์รับเฉพาะผลลัพธ์

2. คำนวณคุณสมบัติผู้ใช้

SDK รวบรวมคุณสมบัติผู้ใช้ปัจจุบัน (device ID, ชนิดสมาชิก, organization UUID ฯลฯ) เพื่อกำหนดว่าผู้ใช้คนนั้นอยู่ในกลุ่มใดของการทดลองใด

3. แคชในเครื่อง

ผลลัพธ์ถูกแคชใน ~/.claude.json ภายใต้ฟิลด์ cachedGrowthBookFeatures อัตราการรีเฟรช: 20 นาทีสำหรับพนักงาน Anthropic, 6 ชั่วโมงสำหรับผู้ใช้ภายนอก

4. ค้นหา flag ในโค้ด

โค้ดธุรกิจสอบถามสถานะฟีเจอร์โดยใช้ชื่อ flag ที่มีคำนำหน้า tengu_* GrowthBook SDK ส่งคืนค่ากลุ่มของผู้ใช้ปัจจุบัน

คุณสมบัติประเภทแหล่งที่มาวัตถุประสงค์

sessionIdstringSession ID ปัจจุบันการทดลองระดับเซสชัน
platformenumprocess.platformแยกตามระบบปฏิบัติการ
organizationUUIDstringข้อมูลรับรอง APIเปิดตัวตามองค์กร
accountUUIDstringข้อมูลรับรอง APIเปิดตัวตามบัญชี
subscriptionTypestringข้อมูลรับรอง APIแยกตาม Free / Pro / Team

วัฒนธรรมการตั้งชื่อ: โลกของ tengu_*

flag ระดับ runtime ทั้งหมดใช้คำนำหน้า tengu_ — "Tengu" (เท็งงุ) เป็นชื่อรหัสภายในของ Claude Code ชื่อ flag ใช้หลักการ สัตว์ / พืช / แร่ธาตุ + คุณศัพท์ ซึ่งตั้งใจให้ไม่โปร่งใส

tengu_kairos — โหมดผู้ช่วย Kairos

สวิตช์ระดับ runtime สำหรับฟีเจอร์ KAIROS ถึงแม้ feature('KAIROS') ระดับ build-time จะผ่าน ก็ยังต้องให้ flag นี้ตรงด้วยถึงจะเปิดใช้งาน การเกตสองชั้นช่วยให้เปิดตัวฟีเจอร์เป็นระยะได้

tengu_amber_stoat — A/B Testing Explore Agent

ควบคุมพฤติกรรมของ Explore sub-agent ในตัว "Amber stoat" (แมวน้ำอมเบอร์) เป็นชื่อรหัสที่สุ่มขึ้นมา ไม่เกี่ยวข้องกับวัตถุประสงค์ของฟีเจอร์ — ป้องกันการเดาฟีเจอร์จากชื่อ flag

tengu_auto_background_agents — ระบบอัตโนมัติ Agent พื้นหลัง

ควบคุมว่างานบางประเภทจะถูกส่งไปยัง background agent โดยอัตโนมัติหรือไม่ แทนที่จะบล็อกผู้ใช้ในเบื้องหน้า

tengu_onyx_plover — Auto-Dream ความจำพื้นหลัง

ควบคุมฟีเจอร์ "ฝันอัตโนมัติ" — จัดระเบียบและรวมความจำของ Agent ในพื้นหลังเมื่อไม่มีการใช้งาน อีกหนึ่งชื่อรหัสที่ไม่โปร่งใส

tengu_glacier_2xr — พฤติกรรมค้นหาเครื่องมือ

ควบคุมพฤติกรรมของ Tool Search น่าจะเป็นการทดสอบ A/B สำหรับอัลกอริทึมค้นหาหรือกลยุทธ์การจัดอันดับ

tengu_birch_trellis — นโยบายสิทธิ์ Bash

ควบคุมนโยบายการตัดสินสิทธิ์ของ BashTool — อาจกำลังทดสอบกฎสิทธิ์ที่เข้มงวดขึ้นหรือผ่อนคลายลง

tengu_scratch — ฟีเจอร์ Scratchpad

ควบคุมฟีเจอร์ "สมุดร่าง" ทดลอง — อาจให้ AI ใช้พื้นที่ชั่วคราวเมื่อจัดการงานที่ซับซ้อน

tengu_quartz_lantern — กลยุทธ์การคำนวณ Diff

ควบคุมวิธีการคำนวณ diff สำหรับการเขียนและแก้ไขไฟล์ น่าจะกำลังทดสอบ A/B อัลกอริทึม diff ต่าง ๆ ว่าส่งผลต่อประสบการณ์ผู้ใช้อย่างไร

กลไก Override สำหรับ Ant-Only

พนักงาน Anthropic มีสองวิธีในการบายพาสการประเมินระยะไกลของ GrowthBook:

Override ด้วย Environment Variable

# ใช้ได้เฉพาะใน builds ที่ USER_TYPE=ant
CLAUDE_INTERNAL_FC_OVERRIDES='{"tengu_kairos": true}' claude

ส่ง JSON object ผ่าน environment variable CLAUDE_INTERNAL_FC_OVERRIDES เพื่อแทนค่า flag ใด ๆ โดยตรง

Override ด้วย Config UI

ใน internal builds แท็บ Gates ของคำสั่ง /config มีอินเทอร์เฟซจัดการ flag แบบกราฟิก ที่สามารถสลับ GrowthBook flag ใด ๆ ได้แบบเรียลไทม์

การติดตามการทดลอง

GrowthBook มีระบบติดตามการเปิดเผยการทดลองแบบครบวงจร:

  • ทุกครั้งที่ค้นหา flag จะบันทึกเหตุการณ์ exposure
  • รายงานผ่าน GrowthbookExperimentEvent ในรูปแบบ protobuf
  • ประกอบด้วย variation_id (0 = กลุ่มควบคุม, 1+ = กลุ่มทดลอง) และค่า in_experiment
  • ข้อมูลใช้วิเคราะห์ผลกระทบเชิงสาเหตุของฟีเจอร์ต่อพฤติกรรมผู้ใช้

GrowthBook กำลัง ย้ายจาก Statsig — โค้ดเบสยังคงมีเลเยอร์ความเข้ากันได้เช่น checkStatsigFeatureGate_CACHED_MAY_BE_STALE()
